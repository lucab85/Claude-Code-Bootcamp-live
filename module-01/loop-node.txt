Think of Claude as a talented junior engineer: before handing off any task, you write a clear spec (Plan) so there's no ambiguity about what "done" looks like. The junior then writes the code (Implement), and you immediately run the tests to confirm the behavior matches the spec (Test) — not just that it compiles. Next comes Review, where you read every line the junior produced, because a junior who codes fast can also introduce subtle bugs, wrong assumptions, or security holes fast. Only after you've signed off does the work get committed and pushed (Commit), making the change part of the shared record. The whole loop is a trust-but-verify contract: you delegate the typing, but you own the outcome. Skipping Review is the most common failure mode because it mistakes "the tests passed" for "the code is correct," which is exactly how a confident junior ships something plausible-looking that silently breaks production three weeks later.
